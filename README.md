## Introduction
A major challenge in the banking industry is the rapid transition to digital banking. With the volume of online transactions increasing by the day, it becomes more difficult to manage financial discrepancies, data breaches, and system downtime, making robust database architectures a priority for operational stability and regulatory compliance. While user interfaces are widely seen as the main driver of customer satisfaction in banking applications, practice demonstrates that a secure financial experience is built on ACID principles (Atomicity, Consistency, Isolation, and Durability), a set of database properties that guarantee reliable transaction processing, ensuring data integrity even during system failures. 

This project builds on a client’s provided specifications and aims to design a robust relational database for an online banking platform. The primary objective is to accurately manage customer profiles, diverse financial products, and daily transactions while establishing automated logic to flag and track overdue fees to reduce the platform’s financial risk. The report details the methodology and processes carried out using Microsoft SQL Server Management Studio (SSMS) to write the T-SQL statements that construct the system.

## Part 1
Database Design and Normalization
To ensure data integrity, eliminate redundancy, and optimize query performance, the proposed database was normalized up to Third Normal Form (3NF).

•	**First Normal Form (1NF)**: All attributes were reduced to atomic values. For example, the client requirement of “Full Name” was separated into FirstName, MiddleName, and LastName. Furthermore, to prevent multi-valued or comma-separated strings, customer addresses were removed from a single text line and structured logically into a separate entity.

•	**Second Normal Form (2NF)**: The database utilizes surrogate primary keys (e.g., CustomerID, AccountID, TransactionID) with IDENTITY properties across all tables. This ensures that every non-key attribute is fully functionally dependent on the primary key, rather than relying on natural keys like Usernames, which could conceptually change.

•	**Third Normal Form (3NF)**: All transitive dependencies were removed. Two major architectural decisions highlight this:
1.	Addresses: Storing elements like City and PostalCode within the Customers table creates a transitive dependency. By creating a dedicated Addresses table linked via an AddressID foreign key, this violation is resolved while allowing multiple customers to share a single residence without data duplication.
2.	Overdue Fees: Rather than linking OverdueFees to the Accounts table, it is linked directly to the Transactions table via TransactionID. Linking to the account would create a transitive dependency and lose the granular context of which specific scheduled payment was missed. Linking to the transaction ensures precise calculation of days overdue. Additionally, the OutstandingBalance is implemented as a computed column (TotalOwed – TotalRepaid) rather than a static field to permanently prevent update anomalies.

## Entity Breakdown and Data Type and Justifications

•	Addresses & Customers: Utilizes NVARCHAR for text fields to support international characters. DateofBirth uses the DATE type as time is irrelevant here. The Customers entity has a foreign key constraint linking Customers to their addresses by an AddressID.

•	Accounts: Employs CHECK constraints on AccountType, AccountStatus, and ReferenceNumber to enforce domain integrity, ensuring only the client’s approved statuses (e.g., ‘Active’, ‘Frozen’) can be entered, and that the ReferenceNumber is only required for Loan and Credit. A UNIQUE constraint should also be employed on the ReferenceNumber attribute, but the attribute will have NULL values for account types that are not ‘Loan’ or ‘Credit Card’, and that will throw an error for Microsoft SSMS, so instead we employ a Unique filtered Index on the ReferenceNumber  attribute that tells the database to ignore the NULLs and focus on real values.

•	Transactions: Financial values (TransactionAmount) use the DECIMAL(18,2) data type to guarantee mathematical precision and prevent floating-point rounding errors. TransactionDate uses DATETIME to capture the exact moment of processing. DueDate and CompletionDate are designated as DATE types to track scheduled payment deadlines for transactions belonging to ‘Loan’ or ‘Credit Card’ accounts. Both columns are intentionally nullable to allow cases where a transaction isn’t a Loan’ or ‘Credit Card’ transaction (for due date), and if a customer is yet to pay complete their loan or credit card payment (for completion date). The Transactions table is linked to the Accounts table with a foreign key constraint, to be able to track accounts that the transactions have been made from.

•	OverdueFees: OverdueFees records and tracks the penalties applied to late payments, fulfilling the client’s specifications. To determine the exact penalty, the system calculates the TotalOwed automatically as a function of DaysOverDue(will be discussed in detail later). Additionally, OutstandingBalance is implemented as a computed column to accurately reflect the customer’s debt. For all financial values such as TotalOwed, and TotalRepaid, the DECIMAL(18,2) data type to guarantee mathematical precision and prevent floating-point rounding errors. Finally, OverdueFees utilizes a foreign key constraint linking it directly to the Transactions table, to track transactions that the fees are attached to.

•	Repayments: The Repayments table is designed to record and track individual payments made by customers to clear their penalty charges. It operates on a one-to-many relationship with the OverdueFees table, allowing a single fee to be paid off across multiple transactions without duplication. Each record logs the specific payment amount, the method used, controlled with a CHECK constraint, and the timestamp of the transaction.

Part 2
Search Accounts by Name
To fulfil the requirement of searching bank accounts by name, a stored procedure (sp_SearchAccountsByName) was created. The stored procedure selects AccountID, CustomerID, AccountName, AccountType, AccountStatus, OpeningDate, ReferenceDate from the Accounts table. The main functionality used to retrieve account names is the LIKE operator paired with the wildcard character (%). A variable @SearchTerm was declared to allow users pass input to the function, which is matched with account names that have that term, and the selected rows of the account are returned. The results are ordered using ORDER BY OpeningDate DESC to satisfy the business rule of showing the most recently opened accounts first.

To execute the procedure, we write a query with the EXEC function, the name of the procedure, and the name we’re looking for. Let’s search for any account name that has ‘Checking’ in it.

Return Upcoming Loan/ Credit payments due in 5 days
To identify loan or credit payments due in less than 5 days, a stored procedure(sp_GetUpcomingPayments) was created. The query uses INNER JOIN on the transactions table to the accounts table to get the account linked to the transactions and uses INNER JOIN on the accounts table to the customers table to get the customers linked to the accounts of the transaction, since the transactions table doesn’t have all the information. Next the rows are filtered specifically for ‘Loan’ and ‘Credit Card’ account types, checks that the Completion date of the transaction is NULL (signifying a customer hasn’t paid). The GETDATE() and DATEADD() functions are used dynamically alongside CAST to accurately calculate the 5-day window.

Insert a new bank Customer
Customers have been added to the bank by manually inserting rows into the Addresses, Customers, and Accounts Separately, but they are normally done as a single process. Insertion of new customers is managed by the AddCustomer stored procedure that incorporates database security and ACID principles while adding customers to the database. The procedure is wrapped in a TRY…CATCH block, to catch errors, and in the BEGIN, and COMMIT TRANSACTION to make sure if there are any errors, the whole Transaction fails maintaining ACID. The banking platform requires users to have a username and password to access their profiles, and we store that in the database, rather than storing plain text passwords, the system uses the built-in HASHBYTES function with the SHA2_512 algorithm, to hash the passwords. 

A unique cryptography salt is generated for every user now of insertion using the NEWID() function. This salt is appended to the plaintext password before hashing and stored in the database, ensuring that identical passwords result in entirely unique hash signatures. With this procedure, we can seamlessly add a customer to the database without having to separately populate tables. To show the result of this procedure, we add a new customer into the database.

Update customer details
The sp_UpdateCustomerDetails stored procedure was created to allow for the modification of existing customer records. As this procedure involves updating the customer table, is stands the risk of having insertion errors by customers, so we use the TRY…CATCH, and the BEGIN, and COMMIT TRANSACTION block to mitigate that. The parameter allows user details to be NULL, if they don’t want to change that information it will be NULL, and it goes into the next step where customer details are updated. Here the ISNULL function is used, it checks if the current value is NULL, and if it is NULL, it keeps the old information, and if it isn’t, that information gets changed.

View that shows all transactions
To fulfil the requirement of reporting all transactions alongside their associated penalties, the TransactionsReportView was created. The SELECT statement used for this was made to capture rows from three different tables using INNER JOINs and a LEFT JOIN. This view gives a comprehensive financial overview by aggregating transaction details (id, amount, and relevant dates), customer’s details (full name and account name) and penalty metrics (fees charged, amounts repaid, and outstanding balances). From the transactions table, an INNER JOIN on the Accounts table was made to get the account related to any transaction, of course this will not include accounts that have not made any transactions yet, similarly, an INNER JOIN was also made on customers table from the accounts table, to get the customers that have such accounts, this will also not include customers without accounts. To show record of fees in the view, we must use a join, but using the INNER JOIN again will truncate the data showing only customers with fees in their record, so we use LEFT JOIN instead, because with it we can ignore the NULL values for customers without FeeID. Another key feature is the ISNULL function in the SELECT statement, and this is still related to customers that don’t have loan or credit accounts, they don’t have fees attached to their account, this makes the rows for fee charged, fee paid and outstanding balance NULL, and this is where the ISNULL function comes in play, it checks if any of those rows are NULL, and replaces them with 0, preventing reporting errors and increasing readability of the financial dataset.

The results show a total of 11 transaction including both normal, and credit/loan transactions, and we can see that the first 3 transactions have no fee ID, indicating they have no fees attached to their account. The customers of the first 2 transactions made normal transactions, while the third customer had a transaction with the bank, and paid on time, so no fees were attached to their account.

Trigger that automates accounts closure.
The purpose of this question is to automate the closure of accounts, specifically ‘Loan’ or ‘Credit Card’ accounts when the customers have completed ALL the pending payments. For this we refer to the AccountStatus attribute, that we defined earlier for the accounts table, when a customer’s account is closed, the value for this attribute will be ‘Closed’. That can be done manually, but it is a tedious process as there can be multiple transactions for multiple customers to keep track of, this is why for this, a trigger AutoCloseLoanAccount was used to implement this. The trigger works on the transactions table after an update. To optimize the process, the trigger only checks for updates on the CompletionDate column, reducing the number of times the trigger runs, as we only care about the completion date of a transaction. The trigger checks the transactions that have had their completion date updated, and makes it into a table named inserted, we then perform an INNER JOIN on the inserted table to the accounts table to find the accounts associated with those transactions. In this step, 3 filtering steps are carried out, to verify that the accounts we want to update are ‘Loan’ and ‘Credit Card’ accounts, that they are currently ‘ACTIVE’ accounts, and that all the transactions corresponding to that account have been completed, this last condition uses a subquery. The subquery asks for only one row where the completion date of all the transactions of that account is null. ‘AND NOT EXISTS’ checks that if even one exists, it means the user hasn’t completed all their transactions, so don’t update it. Putting all the queries together would be ‘change these accounts account’s status to closed, from the rows that had their completion date just changed but check that the accounts are Loan/Credit Accounts, that it is an active account, and all transactions connected to that account have been completed.

List of customers who paid less than 50% of their overdue fees
A requirement was to identify customers who have paid less that 50% of their accumulated overdue fees. To accomplish this, the HighRiskCustomers views was implemented. The customers details (ID, Name) were selected, the details of the fees they owe (TotalRepaid, TotalOwed, and OutstandingDebt), and the number of transactions the user has a penalty on. This implementation requires information across 4 tables, so a chain of INNER JOINs was made connecting the customer table to the accounts table, the accounts table to the transactions table, and the transaction table to the overdue fees table, to identify the customers associated with overdue fees.
Rather than evaluating individual records in isolation, the view uses a GROUP BY clause to aggregate the data with the customers details. The core business logic is in the HAVING clause, which checks the aggregated totals to flag customers whose total repayments fall below 50%.

Additional Recommendations
To guarantee that the digital banking platform remains secure and structurally stable, the following guidelines should be adopted:

1.	Data Integrity and Concurrency Management
In a financial system with a lot of activity, data integrity is very important. The client should strongly follow the ACID principles (Atomicity, Consistency, Isolation, Durability) for all future database development. To make sure that atomicity is maintained and partial data corruption doesn't happen, all multi-step data changes must still be wrapped in BEGIN TRANSACTION and TRY...CATCH blocks. The client must also make sure that the database's Transaction Isolation Level is at least set to READ COMMITTED. Allowing multiple operations to read uncommitted data (Dirty Reads) in banking systems with a lot of traffic can cause huge financial mistakes. To keep a single version of the truth, you need to use shared locks to stop statements from reading rows that are being changed by other transactions.

2.	Database Security and Access Control
The client must follow the "Defense-in-Depth" security posture and the Principle of Least Privilege (PoLP) to the letter. The current foundational architecture uses the default schema, but it is very important that for production deployment, database objects are logically grouped into custom schemas. For example, you could put all of your customer data into a separate ClientData schema. This sets a clear security boundary, letting administrators quickly give or take away rights from a whole group of objects at once. There must be strict limits on who can directly access the foundational tables. Instead, the client should use Views as security boundaries to make sure that staff only have access to the data subsets they need for their jobs. Database Roles should be used to manage administrative privileges instead of individual user accounts.
Conclusion
This project successfully designed, built, and tested a strong relational database architecture that met the needs of the banking industry. The system gets rid of duplicate data and makes sure that user profiles, credit products, and transaction records for different financial entities are all linked together by normalising the schema to the Third Normal Form (3NF). Advanced database objects get rid of mistakes and make it less likely that people will have to do things by hand. The database goes from being a place to store data to an active engine that lowers risk. The platform is completely secure and ACID compliant because it uses cryptographic salting, Role-Based Access Control, and Transaction Control Language. To protect the institution against catastrophic data loss, a full backup and recovery plan was put in place. This plan uses automated SSMS Maintenance Plans for full and differential backups. This plan also ensures that data will last for a long time and that operations will continue. The platform has shown that it can handle complex financial logic, handle errors gracefully, and work in high-concurrency environments through rigorous QA testing. This gives the client a scalable, resilient, and secure base for their digital banking operations.
